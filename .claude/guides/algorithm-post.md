# 알고리즘 문제풀이 포스트 작성 원칙

알고리즘 문제풀이 카테고리의 글을 작성할 때 따르는 원칙이다. 개발 경험 글(원인 분석 → 결과 → 배움)과 달리, 문제풀이 글은 **정답이 아니라 정답에 도달하는 사고 과정**이 중심이다.

## 최우선 원칙: 억지로 채우지 않는다

아래 요소들을 채우기 위해 **내용을 억지로 늘리거나 거짓된 내용을 넣지 않는다.** 실제로 한 생각, 실제로 배운 것만 쓴다. 어떤 항목에 대해 쓸 진짜 내용이 없다면 그 항목은 쓰지 않는 것이 낫다. 분량과 형식보다 정직함이 우선이다.

## 글에 담아야 할 요소

1. **문제 링크** — 글 상단(본문 첫 부분)에 원문 문제로 이동하는 링크를 건다. 클릭 시 바로 문제 페이지로 이동해야 한다. 로그인 여부는 고려하지 않는다.
2. **알고리즘 종류** — 이 문제가 어떤 알고리즘/기법 유형인지 명시한다 (예: 투 포인터, 접두 곱(prefix product), 이진 탐색, DP, 슬라이딩 윈도우). 본문에서 유형을 밝히고, frontmatter `tags`에도 포함한다.
3. **사고 과정** — 정답 코드만 나열하지 않는다. 어떤 과정을 거쳐 그 코드에 도달했는지가 글의 본체다.
   - 처음 떠올린 접근(브루트포스 등)과 그것이 막힌 이유(시간 초과, 제약 위반, 엣지 케이스)
   - 막힌 지점에서 어떤 관찰/발상 전환으로 다음 단계로 넘어갔는지
   - 문제의 제약 조건이 힌트가 된 경우 그 연결
   - 실제 사고 흐름을 재구성하는 것이지, 그럴듯한 서사를 지어내는 것이 아니다. 처음부터 정답 접근이 보였다면 그렇게 쓴다.
4. **배운 것** — 이 문제를 통해 새로 알게 된 것, 또는 다시 확인한 것을 정리한다.
5. **같은 유형 문제로의 일반화** — 동일 카테고리의 다른 문제를 만났을 때 이 문제에서 배운 것을 어떻게 활용할지 적는다.
   - "이런 신호가 보이면 이 기법을 의심하라"는 식의 인식 패턴
   - 같은 기법이 적용되는 다른 문제를 실제로 아는 경우에만 예시로 든다.
6. **정답 코드는 JavaScript** — 모든 풀이 코드는 JS로 작성한다.

## 권장 글 구조

```
[문제 링크]

## 문제            — 요약 + 예시 입출력 + 핵심 제약
## 첫 접근          — 직관적 풀이와 그것이 막히는 이유
## 핵심 아이디어     — 발상 전환, 왜 이 방법이 통하는지
## 구현            — JS 정답 코드 (단계적 개선이 있으면 이해용 → 최적화 순)
## 복잡도           — 접근별 시간/공간 비교
## 배운 것 & 일반화  — 이 문제의 교훈 + 같은 유형 문제에 적용하는 법
```

섹션 이름은 문제에 맞게 바꿔도 되고, 해당 문제에서 의미 없는 섹션은 생략한다. 다만 **흐름(문제 → 막힘 → 발상 → 코드 → 일반화)**은 유지한다.

## 코드 작성 규칙

- 정답 코드는 제출 가능한 완결된 형태로 싣는다.
- 헷갈리기 쉬운 지점(순서 의존, 인덱스 경계, 초기값 등)은 코드 주석이나 코드 직후 문단으로 짚는다.
- 게시 전에 코드를 실제로 실행해 문제의 예제 입력(가능하면 엣지 케이스 포함)에 대해 정답이 나오는지 확인한다.

## frontmatter 규칙

- `tags`에 알고리즘 유형과 `"algorithm"`, 출처 플랫폼(예: `"leetcode"`), 언어(`"JavaScript"`)를 포함한다.
- `description`은 문제 요약이 아니라 "무엇을 어떤 발상으로 풀었는지"가 드러나게 쓴다.

## 체크리스트

- [ ] 문제 링크가 본문 상단에 있는가
- [ ] 알고리즘 종류가 본문과 tags에 명시됐는가
- [ ] 접근 → 막힘 → 발상 전환의 흐름이 실제 사고 과정을 반영하는가
- [ ] 정답 코드가 JS이고, 예제 입력으로 실행 검증했는가
- [ ] 배운 것이 정리됐는가
- [ ] 같은 유형의 다른 문제에 어떻게 활용할지 적었는가 (진짜 할 말이 있는 경우)
- [ ] 억지로 채운 문단이나 지어낸 내용이 없는가
